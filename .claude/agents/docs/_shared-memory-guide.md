# Persistent Agent Memory — 공통 가이드

이 파일은 에이전트 메모리 시스템의 공통 운영 가이드입니다.
각 에이전트는 자신의 메모리 경로(`MEMORY_PATH`)를 별도로 가집니다.

---

## 메모리 저장 방법 (2단계)

**Step 1** — 개별 파일에 메모리 작성 (프론트매터 포함):

```markdown
---
name: {메모리 이름}
description: {한 줄 설명 — 미래 대화에서 관련성 판단에 사용}
type: {user | feedback | project | reference}
---

{내용 — feedback/project 타입은: 규칙/사실 → **Why:** → **How to apply:** 순으로}
```

**Step 2** — `MEMORY.md` 인덱스에 포인터 추가 (내용 직접 작성 금지, 링크만).

---

## 메모리 타입

| 타입 | 저장 시점 | 활용 |
|------|-----------|------|
| `user` | 사용자 역할·선호·지식 파악 시 | 응답 스타일·깊이 조정 |
| `feedback` | 방식 교정 또는 검증된 접근 확인 시 | 같은 피드백 반복 요청 방지 |
| `project` | 진행 중인 작업·결정·마감 파악 시 | 제안의 맥락·방향 조정 |
| `reference` | 외부 시스템 위치 파악 시 | 정보 조회 경로 안내 |

### feedback 타입 주의
- 교정(`"하지 마"`)뿐 아니라 **검증된 접근**(`"맞아, 그렇게 해"`)도 저장
- `**Why:**` + `**How to apply:**` 라인 필수

---

## 저장하지 않을 것

- 코드 패턴·컨벤션·파일 경로 — 코드에서 직접 파생 가능
- git 히스토리·변경 이력 — `git log/blame` 참조
- 디버깅 해결책 — 코드와 커밋 메시지에 존재
- CLAUDE.md에 이미 문서화된 내용
- 현재 대화 내 일시적 작업 상태

---

## 메모리 접근 시점

- 관련 메모리가 있을 것 같을 때
- 이전 대화 작업을 참조하는 것 같을 때
- 사용자가 명시적으로 "기억해줘 / 기억하고 있어?" 요청 시 **반드시** 접근
- 메모리와 현재 코드베이스가 충돌하면 → 현재 코드 신뢰 후 메모리 업데이트

---

## MEMORY.md 관리 규칙

- 200줄 초과 시 잘리므로 **인덱스만** 유지 (내용 직접 작성 금지)
- 주제별로 의미 있게 구성 (시간순 X)
- 중복 전 기존 메모리 업데이트 먼저 확인
- 이 메모리는 버전 관리 공유 범위이므로 프로젝트에 맞게 작성
